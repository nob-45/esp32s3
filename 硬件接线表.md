## 四、摄像头 —— 正点原子 ATK-MC2640-V2.3

**用途**：采集画面 → Wi-Fi JPEG 图传 → **电脑端跑 YOLOv8 做跌倒识别** → 识别结果回传 S3 触发声光报警。
**板子不做本地 AI 推理**（ESP32-S3 跑 YOLO 只有 0.5–1 FPS，不实用），这是**边缘采集 + 服务器 AI**的分布式方案。

### 🔑 XCLK 结论最终版（2026-07-17 二次修正）

**⚠️ 之前上午的"必须外接 XCLK 到 GPIO0"结论是错的，现已推翻。**

经查阅正点原子 **ATK-OV2640 V2.2 原理图** 与 **用户手册 V1.1** 得到铁证：
- 原理图上有独立元件 `Y1 = 24.000 MHz 有源晶振`，`Y1.OUT (pin3) → OV_XCLK → OV2640.PIN13(XCLK)`
- 用户手册第 1 页原文："**集成有源晶振，无需外部提供时钟**"
- 模块 2×9=18pin 排针上**根本没有 XCLK / XVCLK / MCLK 任何丝印**，物理上就没法外接

**正确做法**：
- 代码 `pin_xclk = -1`、`xclk_freq_hz = 0`
- ESP32 **不要**在 GPIO0 输出任何 PWM 时钟，GPIO0 作为 Strapping 引脚保持空闲
- XCLK 由摄像头板载 24MHz 晶振直接驱动，跟主控无关

### 🔑 那"抓帧 100% 超时"到底是什么问题？

重新排查方向（按可能性排序）：

1. **PWDN 引脚悬空 = 芯片掉电**  
   → PWDN 必须接 GND，DVP 才会输出。99% 的抓帧问题都是这个。
2. **RST 引脚悬空 = 一直复位**  
   → RST 必须接 3.3V，或由 GPIO 输出高电平。
3. **VCC 供电不稳 / 电流不足**  
   → OV2640 峰值 100mA，DevKit 3.3V LDO 500mA 要分给整块板，同时供 LCD/雷达时可能掉压。
   → 用外部 3.3V 稳压电源单独给摄像头供电测试。
4. **DVP 数据线走线过长/接触不良**  
   → PCLK/D0-D7 是高速并行信号，杜邦线务必 <10cm 且插实。
5. **FLASH 引脚的 R2/R3 焊接位置**（少见）  
   → 某些批次通过 R2/R3 选择 FLASH 由"外部控制"还是"OV2640 内部 STROBE"，焊错会干扰内部时序。
6. **PCLK 分频未生效**（罕见）  
   → esp32-camera 组件默认会通过寄存器 0xD3/0x11 分频，一般不需要干预。

### ⚠️ 引脚重排原因
正点原子官方推荐引脚跟本项目 LCD/水/雷达/MQ-135 严重冲突（GPIO 5/6/7/10/11/12/13/16/17/18 都被占了），
所以下面这套引脚是**重新分配、避开所有已用 GPIO 的方案**。

|---------|------|------|
| **GND** | ESP32 GND | 地 |
| **VCC** | ESP32 3.3V | ⚠️ 只能 3.3V！接 5V 直接烧 |
| **SCL** | **GPIO3**  | SCCB 时钟（I2C 配置寄存器） |

| **RST**  | **ESP32 GPIO19** ⭐ | ⚠️ **改由 GPIO19 控制，主控启动时发一次复位脉冲（低→高），解决 PID=0 挂死** |

> ✅ **XCLK 不用接！** 正点原子 ATK-MC2640-V2.3 板载 24MHz 有源晶振（原理图 Y1），排针也没有 XCLK 丝印，代码 `pin_xclk = -1`、`xclk_freq_hz = 0`。

### 📌 遇到 `Camera probe failed 0x105 (ESP_ERR_NOT_FOUND)` 请按顺序排查：

1. **PWDN 是否接 GND？** —— 99% 的初次调试失败都是这个！
2. **RST 是否接 3.3V？** —— 悬空会随机复位导致 SCCB 无响应
3. **SDA(GPIO4)/SCL(GPIO3) 是否接对、有没有虚焊？** —— 正点原子模块通常自带上拉，不用额外加电阻
4. **VCC 是否是 3.3V？（不是 5V！）**
5. **共地：ESP32 的 GND 和摄像头模块的 GND 必须至少有一根线相连**
6. **FPC 排线插紧**（若模块是分离式 FPC 座）、**杜邦线插实**（若是排针式）

### 📌 遇到"SCCB 通但抓帧超时"请按顺序排查：

1. **PWDN 引脚是否接到 GND？** ⭐ 头号根因，悬空=芯片掉电=DVP 不吐数据
2. **RST 引脚是否接到 3.3V？** 悬空/低电平=OV2640 一直复位
3. 确认代码是最新版：`pin_xclk=-1`、`xclk_freq_hz=0`、`fb_count=2`、`grab_mode=LATEST`、`frame_size=QVGA`、`jpeg_quality=12`
4. 用短杜邦线（<10cm），避免 PCLK/D0-D7 长线导致的高速信号干扰
5. 摄像头 VCC 3.3V 电流应至少能带 100mA，电源纹波要小；必要时用外部 3.3V 单独供摄像头
6. 音频 I2S 信号线远离摄像头 DVP 数据线（别贴着走）

## 五、音频子系统 —— 功放 + 麦克风（I2S 数字音频）

**用途**：
- **功放（MAX98357A / PAM8302 类 I2S 数字功放）**：语音播报"检测到跌倒，请立即查看"、报警提示音
- **麦克风（INMP441 / ICS-43434 类 I2S 数字麦）**：老人语音求助采集，为后续离线关键词唤醒（"救命/开灯/关火"）预留通道

### 5.1 GPIO 分配（避开已用引脚）

**⚠️ 需要说明的取舍**：ESP32-S3 已被 LCD/摄像头/雷达/水传感器占用了大部分 GPIO，音频 6 根信号线只能借用如下几组"未启用预留位"：

| 分组 | 借用来源 | 是否影响其他功能 |
|------|---------|-----------------|
| GPIO1、GPIO38、GPIO39 | 板子未使用的通用 GPIO | ✅ 无影响 |
| GPIO17、GPIO18 | **雷达玩法B（UART）的预留位** | ⚠️ 若将来启用雷达 UART，需要跟音频二选一 |
| GPIO44 | **UART0 RX**（下载/调试串口的接收端） | ⚠️ 影响电脑向板子发指令，但 `idf.py monitor` 只看板子 TX 输出仍正常 |

### 5.2 功放 MAX98357A（3 根信号线 + 电源共 5 线）

| 功放引脚 | ESP32-S3 GPIO | 说明 |
|---------|--------------|------|
| **VIN** | **5V** | 供电（3–5V 都行，5V 音量更大） |
| **GND** | GND | 地 |
| **BCLK** | **GPIO38** | I2S0 位时钟 |
| **LRC**  | **GPIO39** | I2S0 左右声道选择 (WS) |
| **DIN**  | **GPIO1**  | I2S0 数据输入（S3 → 功放） |
| SD | 悬空 或 3.3V | 关断脚，悬空自动=开启 |
| GAIN | 悬空 | 默认 9dB 增益 |
| 喇叭+ / 喇叭- | 接 4Ω/8Ω 3W 小喇叭 | 直接推动喇叭 |

### 5.3 麦克风 INMP441（3 根信号线 + 电源共 5 线）

| 麦克风引脚 | ESP32-S3 GPIO | 说明 |
|-----------|--------------|------|
| **VDD** | **3.3V** | ⚠️ 只能 3.3V |
| **GND** | GND | 地 |
| **SCK** | **GPIO17** | I2S1 位时钟（借用雷达 UART TX 预留位） |
| **WS**  | **GPIO18** | I2S1 左右声道选择（借用雷达 UART RX 预留位） |
| **SD**  | **GPIO44** | I2S1 数据输出（麦 → S3；借用 UART0 RX） |
| **L/R** | GND | 拉低=左声道；只用单麦时接 GND 即可 |

### 5.4 音频接线要点

1. **两个 I2S 独立走**：ESP32-S3 内部有 **I2S0** 和 **I2S1** 两个控制器，功放走 I2S0（TX 方向）、麦克风走 I2S1（RX 方向），互不干扰
2. **共地**：功放/麦克风 GND 必须与主板 GND 相连，否则会有强底噪
3. **喇叭距离屏幕/摄像头 ≥5cm**：磁铁会干扰摄像头 CMOS，且喇叭震动会让机械外壳共振
4. **麦克风朝向老人常活动区**：INMP441 是全指向数字麦，装在网关正面即可
5. **保留升级路径**：若将来必须启用雷达 UART（玩法B），把音频 I2S1 的 GPIO17/18 挪到其他空闲脚，或与雷达 UART 二选一

### 5.5 音频引脚汇总（复制用）

| 器件 | 信号 | ESP32-S3 GPIO |
|------|------|---------------|
| 功放 MAX98357A | BCLK | **GPIO38** |
| 功放 MAX98357A | LRC (WS) | **GPIO39** |
| 功放 MAX98357A | DIN | **GPIO1** |
| 麦克风 INMP441 | SCK | **GPIO17** |
| 麦克风 INMP441 | WS | **GPIO18** |
| 麦克风 INMP441 | SD | **GPIO44** |

## 六、空气质量传感器 —— MQ-135（模拟 A0 单路输出）

**用途**：室内异味/挥发气体浓度检测（做饭油烟、卫生间氨气、烟气），配合水传感器做"卫生间/厨房异常"分级判定。

| 传感器引脚 | ESP32-S3 引脚 | 说明 |
|-----------|--------------|------|
| **VCC** | **5V** | ⚠️ MQ 系列加热丝需要 5V，接 3.3V 会预热不足、读数虚低 |
| **GND** | GND | 地 |
| **A0**  | **GPIO7** | 模拟输出（ADC1_CH6），读取气体浓度电压 |
| **D0**  | 悬空 | 数字阈值输出，本项目只用 A0 更精细 |

**要点**：
- MQ-135 上电需要 **60 秒预热**（代码里 `AIR_WARMUP_SEC=60`），期间读数会从高向下缓慢收敛，属正常现象
- 预热结束后代码会自动采集 16 轮做"干净空气基线"，此时**周围不要有油烟/香水/酒精**
- 传感器芯片本体（金属网罩内的加热丝）会明显发烫，属正常
- 与 GPIO6（水传感器 A0）共用 ADC1，代码已通过 `adc1_bus.c` 单例统一管理，两者可同时采样

## 七、⚠️ 接线注意事项（务必看！）

1. **雷达必须接 5V**：HLK-LD2410B 标称 5–12V，接 3.3V 会工作不稳或不识别人。
2. **共地**：所有传感器、屏幕、功放、麦克风的 GND 都必须与 ESP32-S3 的 GND 相连，否则：
   - A0 采样值会漂
   - 雷达 OUT 电平会乱跳导致误触发
   - 音频有强底噪甚至无声
3. **GPIO2 的特殊性**：它是 Strapping 引脚，但代码里只做 **输出（背光 PWM）**，不影响启动，放心用。
4. **水传感器不要长时间浸水**：金属电极会电解腐蚀，演示完擦干；长期部署要换成 **电容式** 或 **SUS304 不锈钢探针**。
5. **雷达安装朝向**：正对被监护区域，探测角度约 ±60°，最远约 6m；离墙 ≥30cm 避免多路径反射误判。
6. **接线顺序建议**：先断电插好所有杜邦线 → 检查一遍 3.3V / 5V / GND 有没有插错 → 再插 USB-C 上电。
7. **音频线不要贴着摄像头 D0-D9 走**：I2S 高速时钟会耦合进摄像头并行数据线导致画面雪花，走线尽量分离两侧。

## 八、后续升级：雷达玩法B（UART 读实时距离和能量）

代码里已经预留 UART 接口，届时只需在雷达上额外接两根线：

| 雷达引脚 | ESP32-S3 引脚 | 说明 |
|---------|--------------|------|
| TX | GPIO18 | 雷达 → ESP32 数据 |
| RX | GPIO17 | ESP32 → 雷达 指令 |

⚠️ 这两个脚跟麦克风的 SCK/WS 冲突，**启用雷达 UART 时需要暂停麦克风功能**，或者把麦克风挪到其他空闲脚重新分配。

## 十、上电测试步骤

1. 按上表接好所有线（**先断电！**）
2. 用 USB-C 数据线连 ESP32-S3 到电脑
3. 在项目目录运行：
   ```
   idf.py -p COM3 flash monitor
   ```
   （COM 号根据你电脑实际情况改）
4. 观察 LCD 是否正常显示：
   - 顶部标题「智慧养老网关」
   - PRESENCE 状态（OCCUPIED / EMPTY）随人体走动切换
   - 水位数值、状态条随传感器沾水变化
5. 用手在雷达前 3 秒内挥动 → LCD 应显示 OCCUPIED
6. 用湿棉签轻擦水传感器电极 → LCD 水位百分比应上升
7. Wi-Fi 连接成功后，电脑访问 `http://<板子IP>/stream` 应能看到摄像头画面
8. 运行 `python tools/pc_yolo_fall.py --ip <板子IP> --show`，模拟躺倒动作 → 板子 LCD 弹出"老人跌倒"报警
9. （音频调通后）跌倒报警时功放应播报语音、麦克风可采集语音求助

祝调试顺利 👍
